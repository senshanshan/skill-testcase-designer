# skill-testcase-designer

这是我整理的一个 Codex Skill，用来帮助设计和梳理 Skill、Agent 能力发布前需要覆盖的测试用例。

它的目标不是简单罗列一堆测试点，而是把一个 Skill 的说明、触发规则、工作流、引用资料、脚本、文件边界和发布风险拆成可以执行、可以验收的测试用例。这样在做 Skill 迭代或准备发布时，可以更清楚地判断：核心能力有没有覆盖到，触发是否准确，异常场景是否考虑完整，安全边界有没有被验证，后续改动是否会引入回归问题。

## 主要作用

- 根据 `SKILL.md`、能力描述、目录结构、引用资料等信息生成结构化测试用例。
- 按 Functional、Trigger、Workflow、Compatibility、Exception、Security、Performance、User Experience、Regression 等维度分类覆盖。
- 区分 P0 到 P3 优先级，方便先验证核心路径和高风险问题。
- 覆盖显式触发、隐式触发、误触发、不触发等 Skill 触发准确性场景。
- 对专业 Agent 增加安全边界、提示注入、上下文保持、文件访问边界、历史回归等检查。
- 输出可执行的测试表，包含输入、前置条件、预期过程、预期结果和通过标准。

## 适用场景

我主要把它用在这些场景里：

- 新写了一个 Skill，需要补一套发布前测试用例。
- 已有 Skill 做了改动，需要检查是否会影响原有能力。
- 需要评估一个 Agent 能力是否足够稳定、可控、可发布。
- 想把零散的测试想法整理成一份更标准的测试表。
- 需要同时关注功能、触发、安全、异常和回归，而不是只测一条正常流程。

## 使用方式

在需要设计测试用例时，把目标 Skill 的 `SKILL.md`、能力说明、相关目录或关键约束交给 Codex，并说明希望生成测试用例。该 Skill 会优先收集已有信息，在信息不足但不影响设计时做保守假设；只有关键条件会改变测试方案时，才需要继续追问。

默认输出会包含覆盖摘要和结构化测试用例表。如果已经指定了格式，会按指定格式生成。

## 设计思路

我希望这份 Skill 更偏实用：测试用例要能真正拿去执行，而不是只写概念。每条用例都应该说明测什么、怎么触发、预期怎么处理、怎样算通过，以及这个风险为什么值得测。

在覆盖范围上，它会优先关注真实发布中容易出问题的地方，包括触发漂移、文件或权限边界、异常输入、提示注入、上下文泄漏、模板格式变化和历史能力回归。这样后续维护 Skill 时，也能有一套相对稳定的检查标准。
